00 — RESYNC THE FOLDER

Flatten the nested directories

Your project ended up nested (HashiCorpQ/HashiCorpQ/…) because the archive wrapped the files in a folder. That's why a script sometimes reports "No such file or directory." This is a local-only cleanup — your remote state is untouched, so nothing can break.

$ ls
.github  bootstrap  docs  live  modules  scripts  README.md  .gitignore
One level only

You should see scripts/ and live/ directly under the repo root — not under another HashiCorpQ/. If a stray root main.tf from the old repo remains, delete it: rm -f main.tf.

Commit the clean structure + the new CI

05 — BUSINESS TESTS

Prove the resources actually work — before you delete

Existence isn't function. These exercises make the resources do their job: store an object, keep versions, refuse a public ACL, and reconcile drift. Frame: staging audit evidence in a governed bucket.

1 · Object round-trip (the bucket does its job)

business test
$ B=webiq-infra-artifacts-715841359129
$ echo "audit-evidence-2026" > /tmp/e.txt
$ aws s3 cp /tmp/e.txt s3://$B/evidence/e.txt
$ aws s3 cp s3://$B/evidence/e.txt -           # prints the content back
audit-evidence-2026

2 · Versioning works (compliance retention)

business test
$ aws s3 cp /tmp/e.txt s3://$B/evidence/e.txt   # upload again
$ aws s3api list-object-versions --bucket $B --prefix evidence/e.txt \
     --query 'Versions[].VersionId' --output text
2 version IDs  => history is preserved

3 · The guardrail refuses a public ACL (security control is real)

business test
$ aws s3api put-object-acl --bucket $B --key evidence/e.txt --acl public-read
AccessDenied  => public access block is doing its job (this is the pass)

4 · Drift detection & reconcile (the IaC promise)

live/infrastructure/foundation
$ aws ssm put-parameter --name /webiq-infra/foundation/greeting \
     --value "changed by hand" --type String --overwrite   # create drift
$ terraform plan
~ aws_ssm_parameter.greeting  value => back to code value  (drift detected)
$ terraform apply     # reconciles reality to code
This is the whole value proposition

You changed AWS out-of-band; Terraform noticed and offered to restore the declared state. In a governed account, that's how unauthorized change gets caught and corrected.

Clean the bucket before destroy

You dropped test objects in the bucket. The learning bucket now has force_destroy = true, so terraform destroy will empty and remove it cleanly. If you ever set that to false (production posture), empty it first: aws s3 rm s3://$B --recursive plus delete versions.

06 — GITHUB ACTIONS (CI)

Ship it: plan on every pull request, no stored keys

Auth is OIDC — GitHub presents a short-lived token, AWS trusts it, and Actions assumes a scoped role. Nothing secret is stored in the repo.

1

Create the OIDC provider + role (one-time)

live/infrastructure/github-oidc
$ cd live/infrastructure/github-oidc
$ terraform init && terraform apply
$ terraform output role_arn
arn:aws:iam::715841359129:role/gha-terraform-plan
2

Tell the repo which role to assume

repo variable
$ gh variable set AWS_TF_ROLE_ARN \
     --body arn:aws:iam::715841359129:role/gha-terraform-plan \
     --repo HashiCorpQ/HashiCorpQ
3

Trigger it

Open a PR that touches any .tf file (or run the workflow manually from the Actions tab). The fmt / validate / plan job runs and posts its plan to the run log.

~/Workspaces/HashiCorpQ
$ git checkout -b ci-test && echo "# ci" >> live/infrastructure/foundation/main.tf
$ git commit -am "test CI" && git push -u origin ci-test
$ gh pr create --fill && gh run watch
Plan-only, by design

The role carries ReadOnly + state access — enough to plan, not to apply. Apply stays a deliberate, local action for now. When you want apply-in-CI, widen the role and add a gated apply job on merge to main.
