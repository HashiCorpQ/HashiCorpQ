Prove the ground is solid before you touch anything

Run the pre-flight script first, every session. It checks tools, versions, AWS identity + account, GitHub, and whether the state bucket exists — then tells you which path to take.

$ ./scripts/00-preflight.sh

  ✓ state bucket exists  => start at the FOUNDATION stack
Read the result, don't skip it

If pre-flight prints FAIL for AWS identity, you're on the wrong profile or your SSO token expired — fix that before Terraform, or you'll just get the "failed to get shared config profile" error. Green across the board is your permission to proceed.

★ THE ROADBLOCK PROTOCOL

When a command fails: stay on-target, waste nothing

A fixed method beats improvising. It keeps the work grounded in evidence — which is exactly what stops an AI (or a tired engineer at 2am) from inventing a fix that doesn't exist.

Freeze the failure

Copy the exact command and the full error, verbatim — never paraphrase. Note terraform version, the provider version, and the profile/account. The literal text is the evidence.

Localize the layer

Every error belongs to one layer. Identify it before touching anything — the table below maps the error signature to the layer and the first command to run.

Reproduce minimally

Re-run the smallest command that fails: aws sts get-caller-identity for auth, terraform validate for HCL, terraform plan for logic. Shrink the surface until the cause is obvious.

Use ground truth, not guesses

Verify against the source, not memory: terraform providers schema -json, the provider registry docs, TF_LOG=DEBUG terraform …, terraform console. If a suggested argument isn't in the schema, it isn't real — reject it.

Change one thing, re-run

Never batch fixes. One change, one re-run, observe. Batching hides which change worked and manufactures new failures.

Hand Claude a tight packet

Paste exactly five things: the command, the full error, the relevant file snippet, the versions, and what changed since it last worked. That's the complete context — more burns tokens, less forces guessing.

Anti-hallucination rule

Prefer proof over prose. A terraform plan that shows the intended change is worth more than any assurance. If a fix can't be shown in validate, plan, or the provider schema, treat it as unverified and ask for the doc link before running it.

Drift drill (do this one by hand)

In the console, toggle off versioning on the artifacts bucket. Then run terraform plan. Terraform will show it wants to set versioning back to Enabled — it detected the drift and will reconcile reality to code. That's the whole value proposition in one command.
